# Signhex Production Setup Guide

This is the primary production deployment runbook for support teams deploying Signhex on-prem.

Use this guide when you need to deploy the source-free production bundle generated by:

```bash
bash scripts/bundle/assemble-runtime-bundle.sh <site-name>
```

This guide assumes:

- no DNS
- raw IPv4 addresses only
- CMS served at `https://<cms-ip>`
- backend and player traffic on `http://<backend-ip>:3000`
- production layout:
  - `production/data/`
  - `production/backend/`
  - `production/cms/`
  - `production/electron/`

## 1. Before You Start

### Required hosts

- `Data VM`
- `Backend VM`
- `CMS guest`
- separate player machines

Recommended topology:

- `Data VM`: Ubuntu Server VM
- `Backend VM`: Ubuntu Server VM
- `CMS guest`: small Ubuntu VM by default, or an unprivileged LXC only when Docker/Compose support is already prepared

### Required inputs

Collect these before you build the bundle:

- `SITE_NAME`
- `CMS_PUBLIC_HOST`
- `BACKEND_PRIVATE_HOST`
- `BACKEND_DEVICE_HOST`
- `DATA_PRIVATE_HOST`
- `BACKEND_IMAGE_REF`
- `BACKEND_IMAGE_ARCHIVE`
- `CMS_BUNDLE_SOURCE`
- `PLAYER_ARTIFACTS_DIR`
- optional provided cert files if you are not using generated CMS TLS

### Required tools on the build machine

Run:

```bash
docker info >/dev/null && echo "Docker OK"
tar --version | head -n 1
openssl version
bash scripts/bundle/assemble-runtime-bundle.sh --help | sed -n '1,80p'
```

Purpose:

- confirms the build machine can create the production runtime bundle

Expected result:

- `Docker OK`
- tar version prints
- openssl version prints
- bundle help prints

Failure hint:

- if Docker is unavailable, do not build a deployment bundle from that machine

## 2. Build The Production Bundle

### Recommended command

Run on the build machine from the repo root:

```bash
export SITE_NAME="site-a"
export CMS_PUBLIC_SCHEME="https"
export CMS_PUBLIC_HOST="10.20.0.30"
export BACKEND_PRIVATE_HOST="10.20.0.20"
export BACKEND_DEVICE_HOST="10.20.0.21"
export DATA_PRIVATE_HOST="10.20.0.10"
export BACKEND_IMAGE_REF="ghcr.io/hexmon/signhex-server:1.2.3"
export BACKEND_IMAGE_ARCHIVE="/artifacts/signhex-server-1.2.3.tar"
export CMS_BUNDLE_SOURCE="/artifacts/signhex-nexus-core-1.2.3.tgz"
export PLAYER_ARTIFACTS_DIR="/artifacts/signage-screen/1.2.3"
export ONPREM_CERT_MODE="generate"

bash scripts/bundle/assemble-runtime-bundle.sh --profile production "$SITE_NAME"
```

Purpose:

- assembles the production runtime bundle from released artifacts
- stages runtime-only production folders
- copies prebuilt Windows and Ubuntu player installers
- generates checksums and verification scripts

Expected result:

- the command ends with `Bundle created at: .../dist/onprem/<site-name>`

Failure hint:

- if the command says required artifacts are missing, verify `BACKEND_IMAGE_ARCHIVE`, `CMS_BUNDLE_SOURCE`, and `PLAYER_ARTIFACTS_DIR`

### When to use the default `all` profile

Use this when you want one bundle containing both QA and production deliverables:

```bash
bash scripts/bundle/assemble-runtime-bundle.sh "$SITE_NAME"
```

The generated production runbook still lives at:

```text
dist/onprem/<site-name>/production/PRODUCTION_SETUP_GUIDE.md
```

## 3. Check The Bundle Before Copying It

Run:

```bash
cd "dist/onprem/$SITE_NAME"
./verify-bundle.sh
find production -maxdepth 2 -type f | sort
```

Purpose:

- verifies the generated bundle contents before shipping them to target guests

Expected result:

- checksum validation succeeds
- you can see:
  - `production/data/`
  - `production/backend/`
  - `production/cms/`
  - `production/electron/`

Failure hint:

- if you see `*.SKIPPED.txt`, the bundle was built with `--skip-docker` and is not deployment-ready

## 4. Copy The Runtime Folders To The Target Guests

Choose a release ID:

```bash
export RELEASE_ID="2026-03-30-r1"
export DEPLOY_USER="support"
export DATA_VM_HOST="10.20.0.10"
export BACKEND_VM_HOST="10.20.0.20"
export CMS_VM_HOST="10.20.0.30"
```

Copy the production folders:

```bash
scp -r "dist/onprem/${SITE_NAME}/production/data" "${DEPLOY_USER}@${DATA_VM_HOST}:/opt/signhex/${SITE_NAME}/releases/${RELEASE_ID}/"
scp -r "dist/onprem/${SITE_NAME}/production/backend" "${DEPLOY_USER}@${BACKEND_VM_HOST}:/opt/signhex/${SITE_NAME}/releases/${RELEASE_ID}/"
scp -r "dist/onprem/${SITE_NAME}/production/cms" "${DEPLOY_USER}@${CMS_VM_HOST}:/opt/signhex/${SITE_NAME}/releases/${RELEASE_ID}/"
```

Purpose:

- copies only runtime assets to the target guests

Expected result:

- target guests receive only the generated folders, not the repository

Failure hint:

- if you copied the repo instead of the bundle folders, stop and clean the target path before deployment

Create the active symlinks:

```bash
ssh "${DEPLOY_USER}@${DATA_VM_HOST}" "mkdir -p /opt/signhex/${SITE_NAME}/releases/${RELEASE_ID} /opt/signhex/${SITE_NAME} && ln -sfn /opt/signhex/${SITE_NAME}/releases/${RELEASE_ID}/data /opt/signhex/${SITE_NAME}/current"
ssh "${DEPLOY_USER}@${BACKEND_VM_HOST}" "mkdir -p /opt/signhex/${SITE_NAME}/releases/${RELEASE_ID} /opt/signhex/${SITE_NAME} && ln -sfn /opt/signhex/${SITE_NAME}/releases/${RELEASE_ID}/backend /opt/signhex/${SITE_NAME}/current"
ssh "${DEPLOY_USER}@${CMS_VM_HOST}" "mkdir -p /opt/signhex/${SITE_NAME}/releases/${RELEASE_ID} /opt/signhex/${SITE_NAME} && ln -sfn /opt/signhex/${SITE_NAME}/releases/${RELEASE_ID}/cms /opt/signhex/${SITE_NAME}/current"
```

## 5. Start Each Tier

### Data VM

Run on the Data VM:

```bash
cd "/opt/signhex/${SITE_NAME}/current"
./load-images.sh
./start.sh
./health-check.sh
docker compose --env-file .env.production ps
```

Expected result:

- PostgreSQL and MinIO are healthy

### Backend VM

Run on the Backend VM:

```bash
cd "/opt/signhex/${SITE_NAME}/current"
ls certs
./load-images.sh
./start.sh
./health-check.sh
docker compose --env-file .env.production ps
curl -fsS "http://127.0.0.1:3000/api/v1/health"
```

Expected result:

- backend health endpoint returns success
- `certs/ca.crt` exists

### CMS guest

Run on the CMS guest:

```bash
cd "/opt/signhex/${SITE_NAME}/current"
ls tls
./load-images.sh
./start.sh
./health-check.sh
docker compose --env-file .env.production ps
curl -fsSI "http://127.0.0.1/"
curl -kfsS "https://127.0.0.1/"
```

Expected result:

- HTTP redirects to HTTPS
- HTTPS loads

## 6. Post-Deploy Validation

### Browser checks

Before first login, import the trust material from:

```text
production/cms/admin-browser/
```

Then open:

```text
https://<cms-ip>
```

Confirm:

- login works
- dashboards load
- API calls succeed
- live socket-driven areas connect

### API checks

Run from the CMS guest:

```bash
curl -kfsS "https://127.0.0.1/api/v1/health"
curl -kfsS -o /dev/null -w "%{http_code}\n" "https://127.0.0.1/docs"
```

Expected result:

- `/api/v1/health` returns success
- `/docs` is not publicly enabled in production

### Player handoff

Give the device team:

```text
production/electron/
```

Minimum validation:

- install Windows or Ubuntu player from `installers/`
- copy `config.example.json`
- keep `runtime.mode` as `production`
- point player to `http://<backend-device-ip>:3000`
- pair the device
- confirm the screen appears in CMS

## 7. Routine Operations

### Restart a tier

Run in the relevant folder:

```bash
./stop.sh
./start.sh
./health-check.sh
```

### Check logs

Run in the relevant folder:

```bash
docker compose --env-file .env.production logs --tail=200
```

### Replace CMS TLS material

Replace these files in the CMS folder:

```text
tls/tls.crt
tls/tls.key
```

Then restart the CMS tier:

```bash
./stop.sh
./start.sh
./health-check.sh
```

### Roll back

On each guest:

```bash
./stop.sh
ln -sfn "/opt/signhex/${SITE_NAME}/releases/<previous-release-id>/<folder>" "/opt/signhex/${SITE_NAME}/current"
./start.sh
./health-check.sh
```

Use:

- `data` on the Data VM
- `backend` on the Backend VM
- `cms` on the CMS guest

## 8. Troubleshooting

### CMS loads but API calls fail

Run on the CMS guest:

```bash
curl -kfsS "https://127.0.0.1/api/v1/health"
docker compose --env-file .env.production logs --tail=200
```

Check:

- backend IP in `nginx/default.conf`
- backend tier health
- firewall between CMS and backend

### Backend is up but devices cannot connect

Run on the Backend VM:

```bash
curl -fsS "http://127.0.0.1:3000/api/v1/health"
ss -ltn | grep 3000
```

Check:

- inbound access from the player network to `3000/tcp`
- player config points to backend IP, not CMS IP

### Bundle is missing real image tar files

Run on the build machine:

```bash
find "dist/onprem/${SITE_NAME}" -name '*.SKIPPED.txt' -print
```

If any files are found:

- rebuild without `--skip-docker`

## 9. Production Quick Start

Use this if you already understand the topology and just need the minimum steps.

```bash
export SITE_NAME="site-a"
export CMS_PUBLIC_SCHEME="https"
export CMS_PUBLIC_HOST="10.20.0.30"
export BACKEND_PRIVATE_HOST="10.20.0.20"
export BACKEND_DEVICE_HOST="10.20.0.21"
export DATA_PRIVATE_HOST="10.20.0.10"
export BACKEND_IMAGE_REF="ghcr.io/hexmon/signhex-server:1.2.3"
export BACKEND_IMAGE_ARCHIVE="/artifacts/signhex-server-1.2.3.tar"
export CMS_BUNDLE_SOURCE="/artifacts/signhex-nexus-core-1.2.3.tgz"
export PLAYER_ARTIFACTS_DIR="/artifacts/signage-screen/1.2.3"
export ONPREM_CERT_MODE="generate"

bash scripts/bundle/assemble-runtime-bundle.sh --profile production "$SITE_NAME"
cd "dist/onprem/$SITE_NAME"
./verify-bundle.sh

scp -r "production/data" support@10.20.0.10:/opt/signhex/site-a/releases/2026-03-30-r1/
scp -r "production/backend" support@10.20.0.20:/opt/signhex/site-a/releases/2026-03-30-r1/
scp -r "production/cms" support@10.20.0.30:/opt/signhex/site-a/releases/2026-03-30-r1/
```

Then on each guest:

```bash
cd /opt/signhex/site-a/releases/2026-03-30-r1/<folder>
./load-images.sh
./start.sh
./health-check.sh
```
